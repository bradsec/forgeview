# Preview grid (thumbnail gallery) - MVP design

Date: 2026-06-12
Status: approved, pre-implementation

## Goal

Browse a folder of 3D files as a grid of rendered thumbnails. Clicking a
thumbnail loads that file into the existing 3D viewer. Opening a folder lands
on this grid (it replaces today's behavior where opening a folder only
populates the tree panel).

## Scope (MVP)

In:
- Full Gallery mode in the main viewport area, toggled with the 3D viewer.
- Thumbnails generated on demand (lazy, on scroll into view).
- Scope toggle: current folder only (default) or recursive (all descendants).
- In-memory, session-only thumbnail cache.
- Click tile = load as preview and switch to 3D view. Hover [+] = add to scene
  (existing multi-model path).

Out (deferred to V2):
- Disk cache that survives restart.
- Virtualization for thousands of files.
- Web Worker / OffscreenCanvas offload.
- Drag-select multi-add.

## Approach

Thumbnails are generated by a single shared, hidden WebGLRenderer driven by a
queue, one model at a time, reading the result back with `toDataURL`. Chosen
over reusing the main viewer canvas (couples to the live render loop) and over
a Web Worker (Three loaders plus Tauri `invoke` in a worker is disproportionate
plumbing for the MVP). Browsers cap WebGL contexts near 16, so the grid must
never mount one canvas per tile; thumbnails are static PNG images.

## Architecture

```
thumbnailService            singleton module, not React
  - 1 hidden WebGLRenderer 256x256, Scene, PerspectiveCamera, lights
  - Map<cacheKey, Entry>    Entry = { status, dataUrl? }
                            status: 'pending' | 'ready' | 'error'
  - cacheKey = `${path}:${mtime}:${size}`
  - queue, concurrency 1
  - request(file): returns a Promise<Entry>; dedups by cacheKey
  - generate(file): read_file_bytes -> parse (loaders) -> fitModelToView
       -> render -> toDataURL('image/png') -> disposeModel -> store -> next

PreviewGrid                 React, rendered in main area when mainView==='grid'
  - resolves the file list for gridFolder + gridScope
  - scope toggle (current | recursive), CSS grid of tiles

GridTile                    React
  - IntersectionObserver: on first visibility -> thumbnailService.request(file)
  - states: skeleton (pending), <img> (ready), placeholder (error)
  - click -> setFile(...) + mainView='3d'
  - hover [+] -> addModel(...)
```

## State (viewerStore additions)

- `mainView: 'grid' | '3d'` - folder open sets `'grid'`; clicking a tile sets
  `'3d'`; a toolbar `[Grid | 3D]` toggle flips between them. The toggle is only
  shown when a folder is loaded.
- `gridScope: 'current' | 'recursive'` - default `'current'`.
- `gridFolder: string | null` - which folder the grid currently displays.
  Updated when a folder is opened, when a tree folder is clicked, or when a
  subfolder tile in the grid is clicked.

App render selection: when a folder is loaded and `mainView==='grid'`, render
`PreviewGrid`; otherwise the existing `filePath || loadedModels.length > 0`
rule selects `Viewer3D` vs `DropZone`.

## Data flow

1. Open Folder -> existing `readDirTree`, set `gridFolder`, `mainView='grid'`,
   `explorerVisible=true`.
2. PreviewGrid builds its file list:
   - current: `readDirTree(gridFolder)` (already returns folders + supported
     files for one level).
   - recursive: new `walkDirRecursive(gridFolder)` returns supported files
     under the folder, flattened. File listing only (eager but cheap);
     thumbnails stay lazy.
3. A tile scrolls into view -> `thumbnailService.request(file)` -> queued ->
   resolves to a dataURL -> tile swaps skeleton for `<img>`.
4. Click tile -> `setFile(path, name, ext, size)` + `mainView='3d'` ->
   existing `Viewer3D` Effect 2 loads it (unchanged).
5. Hover [+] -> `addModel({...})` -> existing multi-model path (unchanged).

## Folders in the grid

- Current-folder mode: subfolder tiles are shown; clicking one sets
  `gridFolder` to that subfolder (descend). File tiles and folder tiles
  coexist, folders first.
- Recursive mode: file tiles only.
- The tree panel continues to work in parallel; clicking a tree folder also
  sets `gridFolder`.

## Error handling

- Each queue item is wrapped in try/catch. On read or parse failure the entry
  is stored as `'error'`; the tile renders a placeholder (format badge + file
  name + muted "preview unavailable").
- The queue always disposes the model and advances, so one bad file never
  stalls generation.
- GLTF/Collada that reference external assets (parsed with base path `''`) may
  thumbnail empty; that renders as a blank-ish tile, acceptable for MVP and
  consistent with the existing single-file limitation.

## Visual / theming

- Tiles use existing CSS variable tokens. Format badge reuses the
  `font-mono` badge style; file size uses `font-mono tabular-nums`.
- Skeleton placeholder honors the shipped `prefers-reduced-motion` rule.
- Offscreen scene background is transparent so thumbnails sit cleanly on
  `--bg-app`. Model uses the current theme `modelColor`.
- One radius system (`rounded`) and the single accent, per the design system.

## Testing

- `walkDirRecursive`: unit (mock `@tauri-apps/plugin-fs`) - filters supported
  extensions, recurses nested folders, returns flattened files.
- cacheKey builder: pure unit - `path:mtime:size`; a changed mtime or size
  yields a new key (invalidation).
- thumbnailService queue: unit with the render step mocked - concurrency 1
  ordering, dedup by cacheKey, and that an erroring item still advances the
  queue and is stored as `'error'`. WebGL is not exercised in jsdom (same
  limit as the existing loader tests).
- GridTile: Testing Library with a mocked service - renders skeleton/ready/
  error states; click calls `setFile` and sets `mainView='3d'`; [+] calls
  `addModel`.

## Risks

- Thumbnail generation cost on large/many files. Mitigated by lazy on-visible
  generation and concurrency 1; bounded per view in current-folder mode.
- Recursive mode on a deep tree produces a long file list with no
  virtualization yet; listing is cheap, but very large grids will mount many
  tiles. Acceptable for MVP; V2 adds virtualization.
- Main-thread parse blocking during generation. Concurrency 1 plus generating
  only visible tiles keeps stalls short.
